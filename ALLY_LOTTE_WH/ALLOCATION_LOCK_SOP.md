# 🔒 儲位配發鎖定系統 - 操作手冊

## 📋 概述

本系統確保倉儲移倉驗收的**完全確定性**：
- ✓ **無重複配發** - 每個儲位僅配一張貼紙
- ✓ **清楚邊界** - 明確起始/結束位置
- ✓ **可追蹤性** - 完整簽核記錄和對照表
- ✓ **防錯機制** - 鎖定後不得變更，確保一致性

---

## 🚀 快速流程

### 1️⃣ 資料準備
```
收到移倉明細 (XLSX)
  ↓
提供空儲位清單 (EmptyLoc)
  ↓
執行轉換器 (convert_with_emptyloc.py)
```

### 2️⃣ 轉換執行
```bash
python3 convert_with_emptyloc.py
```

### 3️⃣ 產出物檢查
轉換完成後產生 **5 個檔案**：

| 檔案 | 用途 |
|------|------|
| `_v8.3匯入.xlsx` | 匯入 v8.3 列印機，含配發儲位 |
| `_含入庫單號.xlsx` | 上傳 Google Sheet，含 orderId |
| `_配發簽核報告.txt` | 詳細統計，分列已配/待補/恆溫 |
| `_配發對照表.csv` | orderId → 儲位快速查詢 |
| `_配發鎖定簽核單.txt` | 列印簽核單據，含邊界確認 |

### 4️⃣ 簽核與鎖定
```
1. 列印「配發鎖定簽核單」
2. 三級簽核：配發人 → 驗收人 → 主管
3. 掃描簽核單存檔
4. 鎖定完成 🔒
```

### 5️⃣ 列印與執行
```
1. 匯入「_v8.3匯入.xlsx」到印機
2. 按車趟分段列印
3. 送倉管按儲位貼標
4. 拍照驗收（photo.html）
```

## 📊 產出物詳解

### A. 配發簽核報告 (`_配發簽核報告.txt`)

**用途**：總覽配發狀況，識別待補位

**內容結構**：
```
✅ 已配發儲位 (ALLOCATED)
   • 按走道列舉，顯示起始/結束
   • 全局摘要：首張~末張 orderId，首個~末個儲位

⚠️  待補位 (PENDING)
   • 數量最小的 27 張貼紙
   • 無儲位分配，需人工補位
   • 列出品號、品名、數量供倉管參考

❄️  恆溫倉 (AIR-CONDITIONED)
   • 135 張無儲位配發（空調倉特殊處理）
   • 列出 orderId、品號供對帳
```

**查看方式**：
```bash
cat 0627_棧板資訊_配發簽核報告.txt
```

---

### B. 配發對照表 (`_配發對照表.csv`)

**用途**：倉管快速查詢每張貼紙的儲位

**欄位**：
```
orderId          | 入庫單號 (lotte20260627-1001)
destLoc          | 儲位位置 (BL212)
sku              | 品號
name             | 品名
qtyPcs           | 數量
trip             | 車趟
alloc_batch_id   | 配發批次（BATCH_00001）
alloc_timestamp  | 配發時間戳
```

**使用範例**：
```bash
# 查詢 lotte20260627-1001 的儲位
grep "lotte20260627-1001" 0627_棧板資訊_配發對照表.csv

# 結果
lotte20260627-1001,BL212,BT0044,...
```

---

### C. 配發鎖定簽核單 (`_配發鎖定簽核單.txt`)

**用途**：列印簽核單據，確認鎖定

**內容**：
```
🔒 配發鎖定狀態
   ✓ 282 張已配儲位
   ✓ 12 個走道（BE~BP）
   ✓ 無重複風險
   ✓ 27 張待補位

📋 配發邊界確認
   首張：lotte20260627-1001 → BE024
   末張：lotte20260627-9040 → BP042

✍️  簽核欄位（需簽名日期）
   配發人、驗收人、主管
```

**簽核步驟**：
1. 列印本單據（A4 紙）
2. **配發人**簽名 + 日期
3. **驗收人**簽名 + 日期
4. **主管**簽名 + 日期
5. 掃描存檔至 Google Drive `/配發簽核記錄/`

## 🔄 配發流程示例 (6/27 訂單)

### 輸入
```
移倉明細      : 444 品項 (309 常溫 + 135 恆溫)
空儲位清單    : 282 格 (BE, BF, BG... BP)
排序規則      : 車趟 > 品號 > 數量
走道優先級    : BK(64) > BJ(43) > BL(42) > ... > BH(7)
```

### 轉換邏輯
```
常溫 309 張 vs 空位 282 格
  → 缺口 27 張 (309 - 282)
  → 按數量小到大識別最小 27 張
  → 其餘 282 張按「大板優先」分配儲位
```

### 輸出
```
✅ 已配發：282 張
   • BE:  9張  (BE024~BE253)
   • BF: 21張  (BF014~BF194)
   • BG: 10張  (BG174~BG264)
   • ... (共12走道)

⚠️  待補位：27 張
   • lotte20260627-7001 (113件, BK0046 蛋黃派)
   • lotte20260627-7007 ( 81件, BK9070 黑巧克力派)
   • ... (其餘25張)

❄️  恆溫倉：135 張
   • lotte20260627aircon-1001~7041
```

### 🔒 鎖定驗證
- [ ] 簽核單已列印並簽名

## 🚨 常見問題

### Q1: 為什麼有 27 張「待補位」？
**A:** 常溫倉有 309 張貼紙，但可用儲位只有 282 格。
系統自動選擇數量最小的 27 張暫不配發，避免浪費大儲位。
倉管可根據實際空位再補配。

### Q2: 如何查詢某張貼紙的儲位？
**A:** 使用「配發對照表.csv」：
```bash
grep "orderId" 0627_棧板資訊_配發對照表.csv
```

### Q3: 能否修改已配發的儲位？
**A:** **不能**。配發一旦鎖定（簽核完成），即為最終狀態。
若需變更，必須重新執行轉換器產生新批次。

### Q4: 恆溫倉為什麼無儲位？
**A:** 恆溫倉(空調)有專門的儲位系統，不納入本次配發。
需另行處理，不影響常溫倉配發。

### Q5: 配發邊界「BE024 → BP042」的含義？
**A:** 
- **BE024** = 第一張貼紙的儲位
- **BP042** = 最後一張貼紙的儲位
- 這個範圍已「鎖定」，不得變更
