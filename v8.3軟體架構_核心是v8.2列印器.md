**重要認知**: v8.3 APP 不是核心，核心是 **v8.2 列印器的貼紙輸出** ← 這決定了現場的一切

---

## 🔄 v8.3 的真實架構流程

```
【前置準備階段】— 這裡是核心！
   ↓
1️⃣ 你提供「綁定表」(187行)
   sourceId | sku | 品名 | 箱數 | 來源儲位 | 入庫單 | 上架儲位
   
2️⃣ v8.2 列印器讀取綁定表
   ├─ 自動產生 187 張貼紙
   │  每張包含:
   │  · SKU + 品名 + 箱數
   │  · 來源單號條碼 (源ID)
   │  · 入庫單號 + 入庫單條碼 ← KEY
   │  · 上架儲位 + 上架條碼
   │  · QR Code (內容: {sourceId, sku, orderId, destLocation})
   │
   └─ 同步產出「入庫單表」(CSV/Google Sheet)
      ├─ 187 行入庫單記錄
      ├─ 每行對應 1 板
      └─ 欄位: orderId | sku | 品名 | 箱數 | destLocation

3️⃣ 樂天貼紙貼上 187 板
   ↓
   
【現場驗收階段】— APP 這時才出場
   ↓
4️⃣ 員工 A/C (同一人)
   · 掃貼紙 QR Code (3秒)
     ↓ 讀取: {sourceId, orderId, sku, destLocation}
   · 按拍照按鈕 (5秒)
     ↓ 相機啟動
   
5️⃣ 員工 B
   · 目視確認商品 (30秒)
     ☑ 品名、規格、箱數相符
   
6️⃣ 員工 A/C 拍照
   ↓ 照片自動綁定 orderId
   ↓ 上傳 Google Drive
   ↓ 記錄到 Google Sheet
   ✅ 完成 1 板
```

## 🎯 軟體棧層次

### **v8.2 列印器 (樂天端)**
```javascript
// 核心：讀綁定表 → 產貼紙 + 產入庫單表

input: LOCATION_BINDING_6_22.csv (187行)
  {
    sourceId: "LOTTE-20260622-Air_con622_1-001",
    sku: "BK9275",
    name: "LOTTE Pepero 杏仁巧克力",
    qtyBox: 80,
    sourceLocation: "C05A0194-0",
    orderId: "20260622-001",    ← 自動生成或來自你的系統
    destLocation: "BK012-A01"
  }

output_1: 187 張貼紙 (PDF/紙本)
  每張包含:
  - 源ID條碼
  - 入庫單號 + 入庫單條碼  ← 這是 orderId
  - 品名 + 箱數
  - 上架儲位 + 條碼
  - QR Code: {sourceId, orderId, sku, destLocation}

output_2: 入庫單表 (Google Sheet / CSV)
  columns: orderId | sku | name | qty | sourceId | destLocation
  rows: 187
  
【v8.2列印器的職責】
✓ 確保 187 張貼紙內容正確無誤
✓ 貼紙上的 QR Code 內容完整可讀
✓ 入庫單號與 orderId 一一對應
✓ 上架儲位綁定正確
```

## ✨ v8.3 的本質

```
v8.3 = 「用照片取代電子條碼驗收」

核心轉移:
  驗收不是「APP讀條碼、查表、判斷」
  而是「員工目視確認 + 拍照記錄」

APP 的角色:
  只是「拍照的觸發器」和「記錄的記本」
  
關鍵是:
  v8.2 列印器要把「入庫單資訊」正確印在貼紙上
  這樣 v8.3 APP 才能「掃 QR → 帶出入庫單 → 拍照」
```
